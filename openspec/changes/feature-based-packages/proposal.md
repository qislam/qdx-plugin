## Why

The `qdx package` command currently outputs all manifests to a flat `manifest/` directory with no way to organize packages by feature. When working on multiple features in parallel, there's no mechanism to track which metadata belongs to which feature, deploy features selectively, or compose multiple features into a single release deployment. This leads to manual manifest management and error-prone deployments.

## What Changes

- Add a `--feature` boolean flag to `sf qdx package` that outputs YAML/XML to `manifest/feature/{name}.yml|xml` instead of `manifest/{name}.yml|xml`
- Feature packages can be individually deployed using existing `--deploy` flag
- Add a new `sf qdx release` command that composes release packages from features
- `sf qdx release {name} --add {feature}` adds a feature to a release
- `sf qdx release {name} --remove {feature}` removes a feature from a release
- Release state is persisted in `manifest/release/{name}.json`
- On every add/remove, the release YAML+XML is regenerated by merging all feature YAMLs (union, deduplicate, sort)
- Shared metadata between features is preserved naturally through deduplication

## Capabilities

### New Capabilities
- `feature-flag`: Boolean flag on `qdx package` that redirects output to `manifest/feature/` subdirectory
- `release-command`: New `sf qdx release` command for composing and managing release packages from features

### Modified Capabilities

## Impact

- `src/commands/qdx/package.ts` — add `--feature` flag, modify output path logic
- `src/commands/qdx/release.ts` — new command file
- `src/utils/util.ts` — extract deduplication/sorting logic for reuse
- `package.json` — oclif topic description update for `qdx release`
- `manifest/feature/` and `manifest/release/` — new directory conventions
